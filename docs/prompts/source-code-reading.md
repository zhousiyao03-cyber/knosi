# 开源项目源码深度阅读 Prompt

## 使用方法

将下方 prompt 中的 `[仓库名/URL]` 替换为目标项目，整段发给 AI 工具（Claude Code / ChatGPT 等）。

- 小型项目（<5k 行）：一次性读完
- 大型项目：先跑第一层和第二层，确认重点模块后再深挖第三层
- 可在末尾追加 `我特别关注 [某方面]，请在这方面多花篇幅` 来定制重点

---

## Prompt

对 **[仓库名/URL]** 进行系统性源码阅读，产出一篇结构化的学习笔记，帮助我深入理解这个项目的设计思路、实现细节和值得借鉴的工程实践。

### 阅读流程

按以下顺序逐层深入，每完成一层先输出阶段小结，再进入下一层：

#### 第一层：项目全貌（5 分钟级）

1. 读 README、CONTRIBUTING、ARCHITECTURE（如有）
2. 读 package.json / Cargo.toml / go.mod 等依赖清单，识别核心依赖和技术选型
3. 画出顶层目录树（2-3 层深度），标注每个目录的职责
4. 回答：这个项目解决什么问题？面向谁？核心 API / 入口在哪？

#### 第二层：架构与数据流（30 分钟级）

1. 找到程序入口（main / index / app），追踪启动流程
2. 识别核心抽象（关键 interface / trait / class / type），画出它们之间的依赖关系
3. 追踪一个最典型的用户操作（如：一次请求、一次渲染、一次构建），完整走通数据流
4. 识别分层策略：哪些是对外 API、哪些是内部模块、边界在哪
5. 回答：架构上最重要的设计决策是什么？为什么这样分层？

#### 第三层：核心模块深挖（1 小时级）

针对 2-3 个最核心 / 最复杂的模块：

1. 逐文件阅读，理解内部实现
2. 标注：巧妙的设计模式、性能优化手法、错误处理策略
3. 找出"为什么这样写而不是更简单的写法"的地方，分析 trade-off
4. 识别代码中的防御性编程、边界处理、并发控制等工程细节

#### 第四层：测试与工程化

1. 测试策略：单元 / 集成 / E2E 的比例和组织方式
2. CI/CD：构建流程、发布流程、版本管理
3. 代码质量：lint 规则、类型系统使用深度、文档覆盖率
4. monorepo / workspace 结构（如适用）

### 输出格式

最终产出一篇 Markdown 文章，结构如下：

```markdown
# [项目名] 源码阅读笔记

## 一句话总结
> 用一句话概括这个项目的本质

## 项目画像
- 解决的问题：
- 目标用户：
- 技术栈：
- 仓库规模：约 X 文件 / X 行代码
- 活跃度：最近 commit 时间、contributor 数量

## 架构概览
（附目录结构图 + 核心模块关系图，用 mermaid 或 ASCII）

## 核心数据流
（追踪一个典型操作的完整路径，附代码引用）

## 难点与亮点（重点部分）

### 🔴 难点 1：[标题]
- 问题是什么
- 他们怎么解决的
- 关键代码位置
- 我的理解 / 疑问

### 🟢 亮点 1：[标题]
- 这个设计好在哪
- 对比常规做法的优势
- 可以迁移到我自己项目的场景

（难点和亮点各列 3-5 个）

## 设计决策清单
| 决策 | 选择 | 备选方案 | 为什么选这个 |
|------|------|----------|-------------|
| 状态管理 | X | Y, Z | 因为... |
| ... | ... | ... | ... |

## 值得偷师的模式
（可直接迁移到自己项目中的具体 pattern，附代码片段）

## 我的疑问
（阅读过程中没想通的点，留待后续研究）

## 延伸阅读
（相关的 blog、RFC、issue 讨论链接）
```

### 阅读原则

- **追踪代码而非猜测**：每个结论必须附带具体文件路径和行号
- **重 Why 轻 What**：不要复述代码做了什么，要分析为什么这样做
- **对比思维**：遇到设计决策时，想想还有什么替代方案，为什么没选
- **关联自身**：始终思考"这个技巧能用在我的项目里吗"
- **诚实标注**：没看懂的地方标注为疑问，不要编造解释
